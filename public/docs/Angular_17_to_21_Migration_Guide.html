Chapter 01

The Architectural Shift: zone.js vs Signals Change Detection

Understand how Angular changes its internal update checks when moving from zone-based change detection to modern signal-driven change detection.

1.1 Comparing the Change Detection Models

In Angular 17, change detection typically depended on Zone.js. Whenever an asynchronous operation resolved, Zone.js would flag the application as dirty, prompting the template engine to re-evaluate properties across the entire component tree.

In Angular 21, the framework allows you to opt out of Zone.js entirely. Change detection is now driven directly by reactive Signal updates, which immediately notify the renderer which specific templates need to be redrawn.

⚔️ Showdown: zone.js vs Zoneless change detection
FeatureZone.js (Angular 17 Default)Zoneless Signals (Angular 21 Standard)
InitializationImports zone.js polyfill (adds ~15KB to browser download).No runtime polyfill needed. Zero zone-related overhead.
Change Detection scopeGlobal. Scans components across the tree on any event.Local. Re-evaluates only templates containing modified signals.
Reactives integrationRequires wrapping asynchronous operations in NgZone.run().Runs updates directly within the Signal reactivity model.
Chapter 02

Refactoring @Input & @Output to input() & output()

Learn how to refactor component properties to use signal-based inputs, outputs, and models.

2.1 Comparison of Decorators vs Signals

Moving from decorator-based properties to signals makes component inputs read-only, which prevents accidental bugs where a component directly mutates its own inputs.

refactoring-inputs.ts
// ==================== ANGULAR 17 DECORATORS ====================
import { Component, Input, Output, EventEmitter } from '@angular/core';

@Component({
  template: '<p>{{ value }}</p>'
})
export class OldComponent {
  @Input() value: string = '';
  @Output() trigger = new EventEmitter<void>();
}

// ==================== ANGULAR 21 SIGNALS ====================
import { Component, input, output } from '@angular/core';

@Component({
  template: '<p>{{ value() }}</p>'
})
export class NewComponent {
  value = input<string>('');
  trigger = output<void>();
}
Chapter 03

Query Upgrades: @ViewChild to viewChild()

Learn how to refactor component and element queries to use modern signal-based queries.

3.1 Query Refactoring

Signal queries (like viewChild()) return a read-only signal containing the query result, replacing older decorator-based queries (like @ViewChild) that required lifecycle hook checks (like ngAfterViewInit).

refactoring-queries.ts
// ==================== ANGULAR 17 VIEWCHILD ====================
import { Component, ViewChild, ElementRef, AfterViewInit } from '@angular/core';

@Component({
  template: '<div #box></div>'
})
export class OldQueryComponent implements AfterViewInit {
  @ViewChild('box') box!: ElementRef<HTMLDivElement>;

  ngAfterViewInit() {
    console.log(this.box.nativeElement);
  }
}

// ==================== ANGULAR 21 SIGNAL QUERIES ====================
import { Component, viewChild, ElementRef, effect } from '@angular/core';

@Component({
  template: '<div #box></div>'
})
export class NewQueryComponent {
  box = viewChild.required<ElementRef<HTMLDivElement>>('box');

  constructor() {
    effect(() => {
      console.log(this.box().nativeElement);
    });
  }
}
Chapter 04

Replacing the async pipe with resource()

Learn how to refactor templates from RxJS observables and the async pipe to the new signal-based Resource API.

4.1 Simplifying Asynchronous Data Fetching

In older Angular versions, fetching asynchronous data required setting up RxJS Observables and subscribing to them in the template using the async pipe. Angular 21 simplifies this by managing async data fetching states directly through the resource() API.

async-pipe-vs-resource.ts
// ==================== ANGULAR 17 RxJS + ASYNC PIPE ====================
import { Component } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Observable } from 'rxjs';

@Component({
  template: '<p>{{ (data$ | async)?.name }}</p>'
})
export class OldDataComponent {
  data$: Observable<any>;
  constructor(private http: HttpClient) {
    this.data$ = this.http.get('https://api.store.com/info');
  }
}

// ==================== ANGULAR 21 RESOURCE API ====================
import { Component, resource } from '@angular/core';

@Component({
  template: '<p>{{ info.value()?.name }}</p>'
})
export class NewDataComponent {
  info = resource({
    loader: async () => {
      const res = await fetch('https://api.store.com/info');
      return res.json();
    }
  });
}
Chapter 05

State Sync: Replacing ngOnChanges with linkedSignal()

Learn how to refactor complex input change handling from the ngOnChanges lifecycle hook to linkedSignal().

5.1 Replacing Lifecycle Hooks

Before version 21, resetting local component state when an input changed required implementing the ngOnChanges lifecycle hook. linkedSignal() replaces this lifecycle hook by managing state synchronization reactively.

refactoring-state-sync.ts
// ==================== ANGULAR 17 ngOnChanges ====================
import { Component, Input, OnChanges, SimpleChanges } from '@angular/core';

@Component({ })
export class OldSyncComponent implements OnChanges {
  @Input() userId!: number;
  selectedPostId: number = 0;

  ngOnChanges(changes: SimpleChanges) {
    if (changes['userId']) {
      this.selectedPostId = 0;
    }
  }
}

// ==================== ANGULAR 21 linkedSignal ====================
import { Component, input, linkedSignal } from '@angular/core';

@Component({ })
export class NewSyncComponent {
  userId = input.required<number>();

  selectedPostId = linkedSignal({
    source: this.userId,
    computation: () => 0
  });
}
