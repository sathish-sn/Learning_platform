Chapter 01

Java Language Fundamentals & Advanced Concurrency

From basic variables and OOP modeling to advanced JVM class loaders, heap structures, garbage collection options, generics PECS rules, and virtual thread models.

1.1 Java Core Syntax & JVM Internals

The JVM compiles code using the ClassLoader subsystem (Bootstrap -> Platform -> Application loaders). It separates memory into stack frames for active thread executions and heap storage for globally reachable object instances. The **JIT Compiler** monitors execution frequency, compiling code blocks exceeding C1/C2 thresholds directly into native instructions.

Basic Syntax & Data Types

Java supports both primitive types (stored directly in stack frames) and wrapper objects (stored on the heap). Primitive operations are faster because they avoid heap allocation overhead and the boxing/unboxing step.

BasicsDemo.java
public class BasicsDemo {
    public static void main(String[] args) {
        int lowerBound = 0;
        int upperBound = 10;
        double scale = 1.5;
        
        // Conditional statement
        if (lowerBound < upperBound) {
            // standard loop structure
            for (int i = lowerBound; i < upperBound; i++) {
                System.out.println("Scaled Value: " + (i * scale));
            }
        }
    }
}

Object-Oriented Programming principles

Java is built on four core OOP principles:

  • Encapsulation: Hiding class state fields behind private keywords and exposing access control through public methods (getters/setters).
  • Inheritance: Allowing classes to inherit properties and methods from a superclass. Java limits classes to single inheritance to avoid the complexity of multiple class inheritance.
  • Polymorphism: Supporting multiple implementations of a single method. Resolved via method overloading (compile-time) or method overriding (runtime via virtual method tables).
  • Abstraction: Hiding details using abstract classes or interfaces, allowing you to define APIs separate from their implementations.
OopConceptsDemo.java
public interface Vehicle {
    void startEngine();
}

public class ElectricCar implements Vehicle {
    private int batteryLevel;

    public ElectricCar(int batteryLevel) {
        this.batteryLevel = batteryLevel;
    }

    @Override
    public void startEngine() {
        if (batteryLevel > 10) {
            System.out.println("Engine started silently. Ready to drive!");
        }
    }
}

1.2 Collection Framework Mechanics

The Java Collections Framework organizes memory structures:

  • ArrayList: Backed by a dynamic array. Default initial capacity is 10. When full, it increases capacity by a growth factor of 1.5.
  • LinkedList: Backed by double-linked nodes, optimized for sequential insertions and deletions.
  • HashMap: Operates as an array of buckets. Each key's hashCode() determines its bucket index. If multiple keys resolve to the same bucket index (a collision), they are stored in a linked list. If the number of items in a bucket exceeds the threshold (TREEIFY_THRESHOLD = 8) and the total map capacity is at least 64, Java converts the linked list into a self-balancing Red-Black Tree to improve lookup times.

1.3 Robust Exception Handling Pipeline

Exceptions are Throwable instances handled using explicit control flows. Using **try-with-resources** ensures that resources implementing the AutoCloseable interface are closed automatically, preventing resource leaks.

ExceptionPipeline.java
import java.io.InputStream;
import java.io.FileInputStream;
import java.io.IOException;

public class ExceptionPipeline {
    public void loadConfig(String path) throws IOException {
        try (InputStream input = new FileInputStream(path)) {
            int data = input.read();
            if (data == -1) {
                throw new IllegalArgumentException("Empty configuration file detected");
            }
        } catch (IOException | IllegalArgumentException e) {
            System.err.println("Execution failed: " + e.getMessage());
            throw e;
        }
    }
}

1.4 Generics: Type Erasure & Bounded Wildcards

Generics ensure type safety during compilation. At runtime, the JVM uses **Type Erasure** to remove generic type parameters, replacing them with their upper bounds or Object to maintain backward compatibility with older bytecode versions.

  • Covariance (? extends T): Used to read items from a collection. You can retrieve elements as type T, but you cannot write to the collection.
  • Contravariance (? super T): Used to write items to a collection. You can insert elements of type T, but elements read from the collection are returned as type Object.

1.5 Concurrency: Locks, Synchronization, and Virtual Threads

The Java Memory Model defines how threads interact through shared memory. Key features include:

  • volatile: Forces threads to read and write variables directly from main memory instead of caching them in CPU registers, ensuring visibility across threads.
  • Synchronized Blocks & Lock API: Enforce mutual exclusion. ReentrantLock provides advanced lock control, including fair locking, interruptible lock acquisition, and try-lock timeouts.
  • Virtual Threads: Introduced in Java 21, virtual threads are lightweight threads managed by the JVM instead of the OS. When a virtual thread encounters a blocking operation (like network I/O), the JVM schedules other work on the underlying carrier thread, allowing applications to scale efficiently.
LoomVirtualExecutor.java
import java.util.concurrent.Executors;

public class LoomVirtualExecutor {
    public void runParallelTasks() {
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            for (int i = 0; i < 1000; i++) {
                final int taskId = i;
                executor.submit(() -> {
                    Thread.sleep(java.time.Duration.ofMillis(50));
                    System.out.println("Virtual Task finished: " + taskId);
                    return taskId;
                });
            }
        }
    }
}

2.1 Spring IoC, Bean Scopes & Lifecycle Hooks

The Spring Container lifecycle manages application beans through specific execution phases:

⚔️ Dependency Injection: Field vs. Setter vs. Constructor Injection
Injection TypeSyntax ExamplePros & Cons
Field Injection@Autowired private Service s;Simple syntax, but hides dependencies and makes unit testing difficult (requires reflection to mock). Unrecommended.
Setter Injection@Autowired public void setService(...)Good for optional dependencies, but leaves components open to re-injection bugs.
Constructor Injectionpublic Controller(Service s) { ... }Best practice. Allows dependencies to be declared as final, ensuring immutability and simplifying testing.

The Spring Bean Lifecycle Phases

Spring manages bean creation and destruction through a series of lifecycle phases:

  1. Instantiation: The bean is created in memory.
  2. Populate Properties: Dependencies are injected.
  3. Aware Checks: Spring checks if the bean implements interfaces like BeanNameAware or ApplicationContextAware and sets the appropriate context.
  4. BeanPostProcessor (Before Initialization): Pre-initialization modification hooks run.
  5. Initialization: Methods annotated with @PostConstruct or declared in InitializingBean.afterPropertiesSet() run.
  6. BeanPostProcessor (After Initialization): Post-initialization hooks run. Dynamic proxies are created here to handle features like @Transactional and @Async annotations.
  7. Destruction: When the application context closes, methods annotated with @PreDestroy or declared in DisposableBean.destroy() run.

2.2 REST Controller Routing & Exception Advising

Spring MVC routes HTTP requests to controller methods. Use @RestControllerAdvice to catch validation and business exceptions globally, formatting them into consistent HTTP response payloads.

GlobalExceptionHandler.java
import org.springframework.web.bind.annotation.RestControllerAdvice;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.http.ResponseEntity;
import org.springframework.http.HttpStatus;

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<String> handleValidationError(IllegalArgumentException ex) {
        return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(ex.getMessage());
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<String> handleGenericError(Exception ex) {
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body("An internal error occurred");
    }
}

2.3 Spring Security Filtering Chain

Configure custom security filters to intercept incoming requests and validate authorization tokens before requests reach controller endpoints.

SecurityFilterChainConfig.java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.web.SecurityFilterChain;

@Configuration
public class SecurityFilterChainConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http.csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .anyRequest().authenticated()
            );
        return http.build();
    }
}

2.4 Spring Data JPA: Entity Lifecycle & Lazy Loading

Hibernate manages entities within a persistence context using four entity states:

  • Transient: The entity is created in memory but is not associated with a database session or primary key.
  • Managed: The entity is associated with an active database session. Changes to the entity's properties are automatically synchronized with the database during transaction commit (dirty checking).
  • Detached: The database session is closed or cleared, but the entity still holds data and its primary key.
  • Removed: The entity is marked for deletion and will be deleted from the database during transaction commit.

To prevent the **N+1 query problem**, use JPQL join fetch operations to load parent and child associations in a single query.

Relational Persistence with PostgreSQL

Examine database structures, SQL join types, index optimization, MVCC updates, and table partitioning.

3.1 SQL CRUD & Joins Comparison

Relational operations rely on index design and join optimizations:

Join TypeExecution LogicUse Case
INNER JOINMatches rows based on shared join keys. Non-matching rows are discarded.Fetching associated records (e.g., orders matching customers).
LEFT OUTER JOINReturns all rows from the left table, padding right-side columns with nulls where no match exists.Fetching all records from a primary table regardless of whether they have matching child records.
FULL OUTER JOINReturns all rows from both tables, padding columns with nulls where no match exists on either side.Data reconciliation or auditing.
CROSS JOINProduces a Cartesian product of both tables.Generating combinations of attributes (e.g., sizes and colors).

3.2 Indexing: B-Tree, GIN, and Partial Indexes

PostgreSQL supports several index types, each suited for different use cases:

  • B-Tree Indexes (Default): Organized as self-balancing search trees. Ideal for equality and range queries.
  • GIN (Generalized Inverted Index): Maps values (like arrays or JSON documents) to their parent rows. Ideal for multi-value search and JSONB data.
  • Partial Index: Built only over a subset of rows defined by a filter condition, reducing index size and overhead.

3.3 PostgreSQL MVCC Engine & Vacuuming

PostgreSQL manages concurrent transactions using Multi-Version Concurrency Control (MVCC). Modifications create new versions of rows (tuples) tagged with transaction IDs (xmin and xmax). A background **VACUUM** process cleans up older, unreachable tuple versions to prevent table bloat and reclaim storage.

3.4 Transaction Isolation Levels & Concurrency Anomalies

PostgreSQL provides transaction isolation levels to prevent data anomalies during concurrent operations:

  • Dirty Read: A transaction reads uncommitted changes made by another transaction. (Prevented in PostgreSQL even at the Read Committed level).
  • Non-Repeatable Read: A transaction reads the same row twice and gets different values because another transaction committed changes in the meantime.
  • Phantom Read: A transaction runs a query twice and gets different sets of rows because another transaction inserted or deleted records in the meantime.
Chapter 04

Event-Driven Messaging with RabbitMQ

Learn about message queues, exchange routing topologies, dead-letter exchanges (DLX), and consumer reliability patterns.

4.1 Messaging Core Concepts

RabbitMQ is a message broker that routes messages from producers to consumers using exchanges and queues.

  • Producer: Publishes messages to exchanges.
  • Exchange: Receives messages and routes them to queues based on routing keys and bindings.
  • Queue: Stores messages until consumers are ready to process them.
  • Consumer: Subscribes to queues and processes incoming messages.

4.2 Exchange Types & Routing Keys

The exchange type determines how routing keys map to queues:

  • Direct Exchange: Routes messages to queues matching the exact routing key.
  • Fanout Exchange: Ignores routing keys and broadcasts messages to all bound queues.
  • Topic Exchange: Matches routing keys to queues using wildcards (* matches one word, # matches zero or more words).

4.3 Delivery Guarantees: publisher confirms & Manual consumer ACKs

To prevent message loss if a broker crashes, exchanges and queues should be declared as **durable**, and messages marked as **persistent**. Messages that fail processing are routed to a **Dead-Letter Exchange (DLX)** for error handling or manual review.

NotificationListener.java
import org.springframework.amqp.rabbit.annotation.RabbitListener;
import org.springframework.amqp.support.AmqpHeaders;
import org.springframework.messaging.handler.annotation.Header;
import com.rabbitmq.client.Channel;
import java.io.IOException;

public class NotificationListener {

    @RabbitListener(queues = "sms.notifications", ackMode = "MANUAL")
    public void onMessage(String message, Channel channel, @Header(AmqpHeaders.DELIVERY_TAG) long tag) 
            throws IOException {
        try {
            // Process message
            channel.basicAck(tag, false);
        } catch (Exception e) {
            // Route message to Dead Letter Exchange (DLX) by passing requeue=false
            channel.basicNack(tag, false, false);
        }
    }
}
Chapter 05

In-Memory Operations with Redis Cache

Learn about data structure selection, cache eviction strategies, Sentinel/Cluster replication, caching patterns, and distributed locks.

5.1 Redis Data Structures & Use Cases

Redis is an in-memory data structure store supporting various formats:

  • String: Simple key-value pairs (up to 512MB). Ideal for page templates or session tokens.
  • Hash: Key-value maps. Ideal for storing objects with multiple fields (like user profiles).
  • Sorted Set (ZSET): Unique elements sorted by a score. Ideal for leaderboards or rate limiters.

5.2 Caching Strategies & Performance Anomalies

Understand the trade-offs of different caching strategies:

  • Cache-Aside: The application queries the cache. On a cache miss, it reads from the database, updates the cache, and returns the data.
  • Write-Through: The application updates the cache first, and the cache synchronously updates the database.
  • Write-Behind (Write-Back): The application updates the cache first, and the cache asynchronously updates the database in background batches.

Performance Anomalies

  • Cache Stampede (Thundering Herd): Occurs when a high-traffic cache key expires, causing multiple concurrent requests to hit the database simultaneously. Resolving this requires implementing locking or background renewal.
  • Cache Penetration: Occurs when requests target keys that do not exist in either the cache or the database. To prevent this, cache empty results with a short TTL or use a bloom filter.
  • Cache Avalanche: Occurs when multiple cache keys expire at the same time, causing a spike in database traffic. To prevent this, add random jitter to TTL expiration values.

5.3 Distributed Locking with Lua Scripting

Distributed locks coordinate access to shared resources across multiple application instances. A common implementation strategy uses the SETNX command (setting a key only if it does not exist) combined with an expiration lease time to prevent locks from hanging if a client crashes.

lock.lua
-- Safely release the lock only if the request ID matches
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
Chapter 06

Structural Architecture & Design Patterns

Learn how to implement structural design patterns like CQRS, the Outbox Pattern, and Circuit Breakers in enterprise applications.

6.1 Gang of Four (GoF) Core Patterns

Design patterns provide reusable templates to address common software design challenges:

  • Builder: Separates object construction logic from its representation, supporting step-by-step assembly of complex configurations.
  • Singleton: Restricts instantiation of a class to a single object instance. To ensure thread safety when lazy-initializing a singleton, use double-checked locking with the volatile keyword.
  • Proxy: Intercepts operations to another target object (used in Spring AOP to execute transaction boundaries or security checks).
  • Strategy: Encapsulates interchangeable algorithm families within distinct classes, selecting the appropriate implementation at runtime.

6.2 Distributed Patterns: CQRS, Saga, and Transactional Outbox

Microservices require specialized architectural patterns to maintain data consistency and fault tolerance:

  • CQRS (Command Query Responsibility Segregation): Separates write operations (commands) from read operations (queries), allowing both paths to be scaled independently.
  • Saga Pattern: Coordinates transactions across multiple independent microservices. Can be implemented using **Orchestration** (a central coordinator service directs the workflow) or **Choreography** (services communicate asynchronously using events).
  • Transactional Outbox: Guarantees reliable message delivery in distributed systems. Instead of publishing messages directly to an external message broker during a database transaction, messages are saved in an **Outbox** table within the same transaction. A separate background process polls this table and publishes messages to the broker, ensuring message delivery even if the broker temporarily goes offline.
OutboxEvent.java
import jakarta.persistence.*;
import java.time.LocalDateTime;

@Entity
@Table(name = "outbox_records")
public class OutboxEvent {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String exchange;
    private String routingKey;
    
    @Lob
    private String messagePayload;
    
    private String deliveryStatus; // PENDING, SENT, FAILED
    private LocalDateTime timestamp;
}

6.3 Circuit Breaker Pattern (Fault Tolerance)

The Circuit Breaker Pattern prevents cascading failures in microservice architectures by temporarily blocking requests to failing downstream services. The circuit has three states:

  • Closed: Requests flow normally. If the failure rate exceeds the threshold, the circuit trips to the **Open** state.
  • Open: Requests fail fast, returning a fallback response immediately without hitting the downstream service.
  • Half-Open: After a cooldown period, the circuit allows a limited number of test requests to pass through. If they succeed, the circuit returns to the **Closed** state; if they fail, it trips back to the **Open** state.
