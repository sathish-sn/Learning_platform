Chapter 01

Introduction & tsconfig.json Setup

Discover the benefits of static typing, how the compiler compiles code down to plain JS, and config parameters inside tsconfig.json.

1.1 What is TypeScript?

TypeScript is an open-source programming language developed and maintained by Microsoft. It is a strict syntactical superset of JavaScript, which means that any valid JavaScript code is also valid TypeScript code. TypeScript adds optional static typing, enabling type safety at compile time, self-documenting code, and highly advanced IDE auto-completion.

Crucially, TypeScript is a compiled language that undergoes type erasure. At runtime, the TypeScript compiler (tsc) strips away all type annotations, interfaces, and decorators, leaving behind clean, standardized JavaScript that can run in any browser or JavaScript engine.

👩‍🏫 Teacher's Analogy: Driving without a Dashboard

Writing raw JavaScript is like driving a car at night with a completely dark dashboard. You can press the gas pedal and the car moves (dynamic execution), but you don't know your speed, fuel level, or if the engine is overheating until the car breaks down on the highway (runtime error). TypeScript acts as a fully functional digital dashboard. Before you even pull out of the driveway (compile time), it flashes warning lights if your fuel is low or a tire is flat, preventing catastrophic failures later.

1.2 Setting Up the Compiler Tools

To use TypeScript, install it globally or as a project dependency using npm:

terminal
# Install globally
npm install -g typescript

# Initialize compiler configuration file (tsconfig.json)
tsc --init

# Compile a file
tsc index.ts

# Watch mode (automatically compiles on save)
tsc --watch

1.3 Deep Dive: tsconfig.json

The tsconfig.json file specifies the root files and compiler options required to compile your project. Understanding these flags is critical for managing how strict TypeScript checks your code.

tsconfig.json
{
  "compilerOptions": {
    "target": "ES2022",                /* Compile to this version of JS */
    "module": "NodeNext",              /* Specify module code generation */
    "moduleResolution": "NodeNext",    /* Determine how modules get resolved */
    
    "strict": true,                    /* Enable all strict type-checking options */
    "noImplicitAny": true,             /* Raise error on expressions with implied 'any' */
    "strictNullChecks": true,          /* Ensure null and undefined aren't assigned to basic types */
    
    "outDir": "./dist",                /* Redirect output structure to this folder */
    "rootDir": "./src",                /* Specify root folder of input files */
    "esModuleInterop": true,           /* Support importing CommonJS modules using ESM syntax */
    "skipLibCheck": true,              /* Skip type checking of declaration files (.d.ts) */
    "forceConsistentCasingInFileNames": true
  }
}
Chapter 02

Primitive, Unknown, Never & Enums

Learn about standard primitives, static array types, tuples, unknown vs any, exhaustiveness checks with never, and compile-time enums.

2.1 Core Primitives, Unknown, and Any

TypeScript includes all standard JavaScript primitive types: string, number, boolean, symbol, null, and undefined. It also introduces special type escape hatches:

  • any: Turns off type checking entirely. Avoid this as it bypasses the type system.
  • unknown: A type-safe counterpart to any. You can assign anything to an unknown variable, but you cannot perform operations on it without narrowing the type first.
👩‍🏫 Teacher's Lesson: The Mystery Box Analogy

Think of any as an open box with a label saying "Ignore security, this is safe." The security guard (compiler) lets it pass through without inspection. This frequently leads to runtime explosions. On the other hand, unknown is a locked steel safe. The security guard says: "You can bring this safe inside, but you are not allowed to open it or interact with its contents until you prove what is inside (type guarding or casting)."

any-vs-unknown.ts
let valueAny: any = "Hello";
valueAny.toUpperCase(); // Allowed without check

let valueUnknown: unknown = "Hello";
// valueUnknown.toUpperCase(); // Error: Object is of type 'unknown'.

// We must narrow the type first
if (typeof valueUnknown === "string") {
  console.log(valueUnknown.toUpperCase()); // HELLO
}

2.2 Exhaustiveness Checks with "never"

The never type represents values that should never occur. This is useful for writing exhaustive checks in switch-case blocks. If a developer adds a new type to a union without updating the switch-case, TypeScript will catch the omission during compilation.

never-exhaustiveness.ts
type Shape = "Circle" | "Square";

function getArea(shape: Shape) {
  switch (shape) {
    case "Circle":
      return 3.14;
    case "Square":
      return 1.0;
    default:
      // If another shape is added to the "Shape" union, this line will trigger a compiler error
      const _exhaustiveCheck: never = shape;
      return _exhaustiveCheck;
  }
}

2.3 Arrays, Tuples, and Readonly

A tuple is an array with a fixed number of elements whose types are known at specific indices.

tuples.ts
// Tuple definition
let userTuple: [number, string, boolean] = [101, "Sathisha", true];

// Readonly array to prevent modifications
const immutableNumbers: readonly number[] = [1, 2, 3];
// immutableNumbers.push(4); // Error: Property 'push' does not exist on type 'readonly number[]'

2.4 Enums vs Const Enums

Enums allow you to define a set of named constants. Standard enum structures generate code at runtime, while const enum structures are optimized out during compilation, and their values are inlined directly where used.

const-enums.ts
const enum Roles {
  Admin = "ADMIN",
  User = "USER"
}

let currentRole = Roles.Admin;
// Compiles down to:
// var currentRole = "ADMIN"; (No helper object code generated!)
⚔️ Showdown: Enums vs Const Enums vs Literal Unions vs Readonly Objects

To define key-value config keys or constants, developers use multiple approaches. Here is the direct code comparison:

constants-showdown.ts
// Method 1: Standard Enum (Generates an overhead JS IIFE object)
enum EnumRoles { Admin = "ADMIN", User = "USER" }

// Method 2: Const Enum (Inlined at compile time - no runtime overhead)
const enum ConstEnumRoles { Admin = "ADMIN", User = "USER" }

// Method 3: Literal Unions (Most flexible, lightweight, idiomatic TypeScript)
type UnionRoles = "ADMIN" | "USER";

// Method 4: Readonly Objects with 'as const' (Best if you need to iterate runtime values)
const ObjectRoles = {
  Admin: "ADMIN",
  User: "USER"
} as const;
type ObjectRoleType = typeof ObjectRoles[keyof typeof ObjectRoles]; // "ADMIN" | "USER"
ApproachProsCons
Standard Enum Supports reverse mapping (e.g. Enum[0]). Creates unnecessary compilation helper code in JavaScript.
Const Enum Zero runtime memory footprint. Cannot be dynamically iterated; has issues with lookup files.
Literal Unions Extremely clean syntax, native compiler speed. Values do not exist at runtime (cannot iterate them).
Readonly Object Values exist at runtime, easy to convert to dynamic arrays. More verbose type recovery syntax (uses typeof).
Chapter 03

Object Types, Interfaces & Aliases

Learn about defining object shapes, index signatures, interface extensions, and interfaces vs type aliases.

3.1 Type Aliases vs Interfaces

In most cases, you can use either interface or type (alias). However, there are key differences:

FeatureInterfaceType Alias
ExtensibilityCan inherit other interfaces using extends.Can combine other types using intersection &.
Declaration MergingYes. Declaring the same interface multiple times merges their properties.No. Duplicate type names throw compiler errors.
Scope CapabilitiesCan only describe object/class shapes.Can represent primitives, unions, intersections, and tuples.
declaration-merging.ts
// Interface merging
interface Client {
  name: string;
}
interface Client {
  id: number;
}

// The compiler merges these into: { name: string; id: number }
const activeClient: Client = { name: "John", id: 404 };
⚔️ Showdown: Interface Extension vs Type Intersection

How should you inherit properties? Using interfaces with the extends keyword or type aliases using the intersection operator (&)? Let's look at both:

extension-vs-intersection.ts
// Method A: Interface Extends
interface BaseUser { id: number; }
interface AdminUser extends BaseUser { permissions: string[]; }

// Method B: Type Intersection
type BaseUserType = { id: number };
type AdminUserType = BaseUserType & { permissions: string[] };

Differences in Error Handling:

  • Interface Extension detects conflicting property overrides at compile-time and throws a clean error:
    interface AdminUser extends BaseUser { id: string; } // ERROR: Interface 'AdminUser' incorrectly extends interface 'BaseUser'.
  • Type Intersection allows conflicts to compile but results in a property type of never, causing bugs later:
    type Admin = BaseUserType & { id: string }; // id compiles to type 'number & string' which yields type 'never'.

3.2 Index Signatures

Index signatures allow you to define types for objects when you do not know all the property names beforehand.

index-signatures.ts
interface Dictionary {
  [key: string]: string; // Any key must map to a string value
}

const translations: Dictionary = {
  hello: "hola",
  bye: "adios"
};
⚔️ Showdown: Index Signatures vs Record<K, T> vs ES6 Map

To implement dynamic lookup tables (dictionaries), you can use three different methods:

dictionaries-showdown.ts
// Method 1: Index Signature (Traditional syntax)
interface LegacyDict { [key: string]: number; }

// Method 2: Record Utility (Clean, functional syntax)
type ModernDict = Record<string, number>;

// Method 3: ES6 Map Class (Strongest types, supports any key type)
const mapDict = new Map<string, number>();
ApproachType Checking BehaviorPerformance Note
Index Signature & Record Allows setting property keys with strings. Note: does not catch missing keys at runtime (reading dict["invalid"] returns undefined but TS infers it as number). Compiles down to plain JS objects, offering fast retrieval.
ES6 Map Ensures that reading values returns number | undefined, forcing you to write check blocks before using values. Has slight memory overhead compared to plain JS objects. Recommended for highly dynamic collections.
Chapter 04

Unions, Intersections & Literal Types

Learn how to combine types using unions and intersections, work with literal types, and write type assertions.

4.1 Union and Intersection Types

Union types (|) allow a value to be one of several types. Intersection types (&) combine multiple types into one, requiring all properties to be present.

unions-intersections.ts
// Union Type
let identifier: string | number;
identifier = "ID123"; // Ok
identifier = 123;     // Ok

// Intersection Type
interface HasName { name: string; }
interface HasAge  { age: number; }

type Employee = HasName & HasAge;

const staff: Employee = {
  name: "Alice",
  age: 30
};

4.2 Constant Literal Assertions

Using as const tells the compiler to treat all object properties as readonly literals, rather than generic types like string or number.

as-const.ts
const config = {
  port: 3000,
  host: "localhost"
} as const;

// config.port = 4000; // Error: Cannot assign to 'port' because it is a read-only property.
⚔️ Showdown: Readonly Modifiers vs Readonly<T> vs Constant Assertions

TypeScript offers different ways to protect properties from write modifications:

readonly-comparison.ts
// Method A: Readonly Property Modifier (Shallow protect for specific keys)
interface Profile {
  readonly userId: number;
  username: string;
}

// Method B: Readonly<T> Utility (Shallow protect for all keys)
type ReadonlyProfile = Readonly<{
  userId: number;
  details: { role: string }
}>;
// Warning: details object is still mutable! (e.g. profile.details.role = "User" is allowed)

// Method C: Constant Assertions (Deep recursive protect for objects)
const systemConfig = {
  server: "production",
  details: { port: 80 }
} as const;
// Deeply protected! (e.g. systemConfig.details.port = 443 throws an error)
Chapter 05

Narrowing & Discriminated Unions

Learn about type narrowing, type predicates, and pattern matching using Discriminated Unions.

5.1 Type Guards and Narrowing

Narrowing is the process where TypeScript refines a variable's type into a more specific one within a block of code, using logical checks like typeof, instanceof, or in.

type-guards.ts
function processInput(input: string | number) {
  if (typeof input === "string") {
    // input is narrowed to string here
    return input.length;
  }
  // input is narrowed to number here
  return input + 10;
}

5.2 User-Defined Type Guards

You can write custom functions that check if a variable matches a type, returning a Type Predicate (like parameter is Type).

custom-guard.ts
interface Bird { fly(): void; }
interface Fish { swim(): void; }

// Custom type guard
function isFish(pet: Bird | Fish): pet is Fish {
  return (pet as Fish).swim !== undefined;
}

function move(pet: Bird | Fish) {
  if (isFish(pet)) {
    pet.swim();
  } else {
    pet.fly();
  }
}

5.3 Discriminated Unions

A discriminated union uses a common literal property (the "discriminant") in each type, allowing you to safely differentiate between them in your code.

discriminated-unions.ts
interface SuccessResponse {
  status: "success"; // Discriminant
  data: string;
}

interface ErrorResponse {
  status: "error";   // Discriminant
  error: string;
}

type ApiResponse = SuccessResponse | ErrorResponse;

function handleResponse(res: ApiResponse) {
  if (res.status === "success") {
    console.log(res.data);
  } else {
    console.log(res.error);
  }
}
⚔️ Showdown: Type Predicates vs Assertion Functions vs Inline Checks

To verify if an object is of a specific type, you can use these three methods:

guarding-showdown.ts
interface Admin { isAdmin: boolean; role: string; }

// Method 1: Type Predicate (Checks type and returns a boolean)
function isAdminUser(user: any): user is Admin {
  return user && user.isAdmin === true;
}

// Method 2: Assertion Function (Throws an error if type check fails)
function assertIsAdmin(user: any): asserts user is Admin {
  if (!user || user.isAdmin !== true) {
    throw new Error("Not an admin!");
  }
}

// Method 3: Inline in-operator check (Zero overhead, native check)
function process(user: any) {
  if (user && "isAdmin" in user) {
     // user is partially narrowed here
  }
}
Chapter 06

Functions & Call Overloads

Learn about typing parameters and return values, writing function overloads, and configuring the "this" context.

6.1 Function Overloading

Function overloads allow you to define multiple call signatures for a single function. You write the signatures first, followed by the actual implementation that handles the logic for all of them.

overloads.ts
// Overload signatures
function combine(a: string, b: string): string;
function combine(a: number, b: number): number;

// Implementation signature (not visible to users calling the function)
function combine(a: any, b: any): any {
  return a + b;
}

console.log(combine(10, 20));    // 30
console.log(combine("A", "B"));  // "AB"
Chapter 07

Class Architecture & OOP

Learn about access modifiers, readonly properties, shorthand parameter properties, abstract classes, and inheritance.

7.1 Access Modifiers and Shorthand Properties

TypeScript provides three modifiers to control class member visibility:

  • public (default): Accessible anywhere.
  • private: Accessible only within the declaring class.
  • protected: Accessible within the declaring class and its subclasses.
class-modifiers.ts
class BankAccount {
  // Parameter properties shorthand: defines access modifiers directly in constructor
  constructor(
    public readonly accountNumber: number,
    protected balance: number,
    private ownerSecret: string
  ) {}

  getBalance() {
    return this.balance;
  }
}

const myAcc = new BankAccount(9901, 5000, "pwd123");
console.log(myAcc.accountNumber);
// console.log(myAcc.balance); // Error: 'balance' is protected

7.2 Abstract Classes

Abstract classes serve as blueprints for other classes. They cannot be instantiated directly and can contain abstract methods that must be implemented by subclasses.

abstract-classes.ts
abstract class Logger {
  abstract log(msg: string): void;

  info(msg: string) {
    this.log(`[INFO] ${msg}`);
  }
}

class ConsoleLogger extends Logger {
  log(msg: string) {
    console.log(msg);
  }
}
⚔️ Showdown: Interface implements vs Abstract Class inheritance

To define class contracts and shared behaviors, you can use these two approaches:

oop-showdown.ts
// Method 1: Interface contract (Pure shape, zero output JavaScript code)
interface Runnable { run(): void; }
class Runner implements Runnable {
  run() { console.log("Running..."); }
}

// Method 2: Abstract Class (Provides both contract and concrete helper methods)
abstract class BaseRunner {
  abstract run(): void;
  warmUp() { console.log("Stretching muscles..."); }
}
class Athlete extends BaseRunner {
  run() { this.warmUp(); console.log("Sprinting!"); }
}
Chapter 08

Generics & Constraints

Learn how to write reusable, type-safe components using generics, generic constraints, and defaults.

8.1 Writing Generic Functions

Generics allow you to write functions or classes that work with multiple types while maintaining type safety, rather than using any.

generics.ts
function identity<T>(arg: T): T {
  return arg;
}

const result1 = identity("myString"); // Type of result1 is "myString" (inferred)
const result2 = identity<number>(123); // Type of result2 is number

8.2 Generic Constraints

You can restrict the types that a generic parameter can accept using the extends keyword.

generic-constraints.ts
interface Lengthwise {
  length: number;
}

function logLength<T extends Lengthwise>(obj: T): number {
  return obj.length;
}

logLength([1, 2, 3]); // Ok (Array has .length)
logLength({ length: 10, value: "test" }); // Ok
// logLength(42); // Error: Argument of type 'number' is not assignable to parameter of type 'Lengthwise'.
Chapter 09

Advanced Type Manipulation

Unlock advanced meta-types using keyof, indexed access, conditional type branching, mapped types, and template literals.

9.1 Keyof and Indexed Access

The keyof operator takes an object type and returns a union of its keys. Indexed access allows you to look up the type of a specific property on another type.

keyof-indexed.ts
interface User {
  id: number;
  username: string;
}

type UserKeys = keyof User; // "id" | "username"
type IdType = User["id"];    // number

9.2 Mapped Types and Conditional Types

Mapped types allow you to create new types based on existing ones by transforming their properties. Conditional types let you select types based on a relation check.

mapped-conditional.ts
// Mapped Type that makes all properties boolean
type OptionsFlags<T> = {
  [Property in keyof T]: boolean;
};

type FeatureOptions = OptionsFlags<{
  darkMode: () => void;
  notifications: () => void;
}>;
// Compiles to: { darkMode: boolean; notifications: boolean }

// Conditional Type
type IsString<T> = T extends string ? true : false;
type Check1 = IsString<string>; // true
type Check2 = IsString<number>; // false

10.1 Key Utility Types

TypeScript provides built-in global utility types to facilitate common type transformations.

Utility TypeBehaviorExample
Partial<T>Makes all properties in T optional.Partial<{a: number}> → {a?: number}
Required<T>Makes all properties in T required.Required<{a?: number}> → {a: number}
Readonly<T>Makes all properties in T read-only.Prevents updating properties on the object.

interface Task {
  id: number;
  title: string;
  completed: boolean;
}

// Partial task for updates
function updateTask(id: number, fieldsToUpdate: Partial<Task>) {
  // ...
}

updateTask(1, { completed: true }); // Valid even without title

11.1 Class and Method Decorators

Decorators are a stage 2 proposal for JavaScript and are available in TypeScript as an experimental feature. They allow you to annotate and modify classes and their members at runtime.

decorators.ts
function Freeze(constructor: Function) {

@Freeze
class ImmutableModel {

Modules, Declarations (.d.ts) & Namespaces

Learn about import/export type configurations, Namespaces, global definitions, and declaration files.

12.1 Importing and Exporting Types

Using the import type syntax tells the compiler to only import the types, ensuring that the import is completely removed from the compiled JavaScript output.

types-import.ts
// Importing only the type interface
import type { User } from "./models";

const activeUser: User = {
  id: 101,
  username: "Sathisha"
};

12.2 Declaration Files (.d.ts)

Declaration files (.d.ts) provide type definitions for existing JavaScript libraries that do not have their own types, without containing any executable code.

global-declarations.d.ts
// Declare global variables or external modules
declare const GLOBAL_API_KEY: string;

declare module "custom-legacy-library" {
  export function processLegacyData(data: any): boolean;
}
Chapter 13

Compilation, Best Practices & Tooling

Learn how to configure build tools, write clean TypeScript, and troubleshoot common compiler issues.

13.1 Production Compilation Best Practices

Critical Production Rules

  • Enable strict mode: Set "strict": true in your tsconfig.json to enforce type safety.
  • Avoid any: Use unknown or generic parameters instead of any to prevent bypasses of the type system.
  • Avoid non-null assertions: Use optional chaining (?.) or explicit null checks instead of the non-null assertion operator (!).
  • Avoid Enums in Library APIs: Consider using Union Types of string literals instead of standard enums to make imports simpler for consumers.
