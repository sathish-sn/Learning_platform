Chapter 01

The Paradigm Shift: Standalone Architecture

Examine how Standalone components remove the need for NgModule declarations, making dependencies explicit at the component level.

1.1 Standalone Components by Default

Historically, Angular organized applications using NgModules to manage compilation contexts and dependency injection boundaries. Starting with Angular 17, Standalone APIs are the default. Components, directives, and pipes are declared with standalone: true in their decorator, bypassing the intermediate module declaration step.

👨‍🏫 Professor's Lecture: Why Modules are Retiring

In older Angular versions, NgModules created a layer of indirection. A component did not explicitly state its dependencies. Instead, it relied on its declaring NgModule's imports. This made tree-shaking difficult because the compiler could not easily prove whether a component actually used an imported library. Standalone components solve this by requiring explicit declarations inside their imports array, making dependency tracking much more precise.

standalone.component.ts
import { Component } from '@angular/core';
import { CommonModule } from '@angular/common';

@Component({
  selector: 'app-profile',
  standalone: true,
  imports: [CommonModule],
  template: `
    <div class="profile">
      <h3>User Profile</h3>
    </div>
  `
})
export class ProfileComponent {}
Chapter 02

The New Built-in Control Flow (@if, @for, @switch)

Understand how the new template block syntax compiles directly to standard JavaScript, replacing traditional structural directives.

2.1 Syntactic Sugar vs Built-in Execution

Angular 17 introduced a new built-in control flow syntax using the @ prefix. This syntax replaces older structural directives like *ngIf, *ngFor, and *ngSwitch. Instead of parsing custom attributes at runtime, the compiler compiles these blocks directly into standard JavaScript instructions during compilation.

control-flow.component.html
<!-- 1. Conditonals (@if, @else) -->
@if (isLoggedIn) {
  <p>Welcome back, user!</p>
} @else {
  <p>Please log in.</p>
}

<!-- 2. Iterators (@for) - require tracking keys natively -->
@for (item of items; track item.id) {
  <li>{{ item.name }}</li>
} @empty {
  <p>No items available in list.</p>
}

<!-- 3. Switches (@switch) -->
@switch (role) {
  @case ('admin') { <span>Admin Dashboard</span> }
  @case ('editor') { <span>Editor Panel</span> }
  @default { <span>Standard View</span> }
}
👨‍🏫 Professor's Analysis: Performance Wins

Built-in control flow improves performance by reducing rendering overhead. The track expression in @for is now checked at compile-time, helping the rendering engine efficiently match elements to DOM nodes during collection changes. This yields up to a 90% rendering speed improvement for large lists compared to legacy structural directives.

Chapter 03

Deferrable Views (@defer) & Performance Tuning

Learn about lazy-loading component blocks using the @defer template control system.

3.1 Progressive Loading in Templates

The @defer block allows you to lazy-load dependencies (like heavy charts or dialog forms) only when a specific trigger condition is met, reducing the initial bundle size of your page.

defer-views.component.html
<!-- Defer loading until user hovers over the placeholder area -->
@defer (on hover) {
  <app-heavy-chart [data]="chartData" />
} @placeholder {
  <div class="skeleton">Hover here to load the chart...</div>
} @loading {
  <div class="spinner">Loading dependency modules...</div>
} @error {
  <p>Failed to load chart bundle.</p>
}

Supported Triggers

  • on idle: Loads when browser enters idle state (default).
  • on viewport: Loads when the placeholder enters the visible screen area.
  • on interaction: Loads when user clicks or presses a key on the placeholder.
  • on hover: Loads when the mouse hovers over the placeholder.
  • on immediate: Loads immediately after the parent template renders.
  • on timer(duration): Loads after a specified delay.
Chapter 04

Introduction to Angular Signals

Master the reactive primitives: writeable signals, computed values, and execution effects.

4.1 Writeable Signals, Computeds, and Effects

Angular 17 stabilized the foundations of **Signals**, a reactive primitive that tracks state changes within your application. Signals offer a cleaner alternative to RXJS for basic state management and can run without the overhead of Zone.js.

signals.component.ts
import { Component, signal, computed, effect } from '@angular/core';

@Component({
  selector: 'app-counter',
  standalone: true,
  template: `
    <button (click)="increment()">Count: {{ count() }}</button>
    <p>Double Count: {{ double() }}</p>
  `
})
export class CounterComponent {
  count = signal(0);
  double = computed(() => this.count() * 2);

  constructor() {
    effect(() => {
      console.log("Current count is: " + this.count());
    });
  }

  increment() {
    this.count.update(prev => prev + 1);
  }
}
Chapter 05

Hydration & Server-Side Rendering (SSR)

Learn about non-destructive hydration, which preserves the server-rendered DOM instead of rebuilding it on the client.

5.1 Non-destructive Hydration

Before version 17, Angular SSR suffered from "destructive hydration." The client-side app would download, discard the server-rendered DOM, and rebuild it from scratch. This caused noticeable page flickering. Angular 17 introduces **Non-destructive Hydration**, which matches the server-rendered DOM nodes to the client-side component representation without tearing down the DOM.

app.config.ts
import { ApplicationConfig } from '@angular/core';
import { provideClientHydration } from '@angular/platform-browser';

export const appConfig: ApplicationConfig = {
  providers: [
    provideClientHydration()
  ]
};
Chapter 06

Application Tooling: esbuild & Vite Integration

Learn about the Application Builder, which integrates esbuild and Vite to compile applications up to 10x faster.

6.1 Modern Compilation Pipelines

Angular 17 introduced a modern build pipeline based on **esbuild** and **Vite**, replacing the legacy Webpack build process. This configuration compiles TypeScript, bundles modules, and minifies assets significantly faster.

Key Tooling Enhancements

  • esbuild for build compilation: Runs parallel compilation steps, drastically reducing local build times.
  • Vite dev server integration: Leverages Vite for hot module replacement during local development.
  • Unified Configuration: Build processes for SSR and client bundles are handled within a single configuration.
